## 3. The Hybrid Quantum‑Classical Workflow: From Idea to Production

A typical quantum‑ready pipeline in 2025 follows these stages:

1. **Problem Formulation** – Classical pre‑processing to encode the problem (e.g., graph, Hamiltonian) into a quantum‑compatible format.
2. **Circuit Generation** – Use SDK (Qiskit, Braket, Q#) to create a parameterized circuit.
3. **Hybrid Optimization Loop** – Classical optimizer (Adam, COBYLA) sends parameters to the quantum runtime; the QPU returns expectation values.
4. **Error‑Mitigation / Correction** – Runtime automatically applies zero‑noise extrapolation or surface‑code cycles for logical qubits.
5. **Post‑Processing & Decision** – Classical code interprets results, feeds them back into business logic (e.g., portfolio rebalancing).

### 3.1 Code Sample – Quantum‑Enhanced Portfolio Optimization (Azure Quantum)

```qsharp
open Microsoft.Quantum.Intrinsic;
open Microsoft.Quantum.Canon;
open Microsoft.Quantum.Diagnostics;

operation QAOA_Portfolio(weights : Double[], gamma : Double, beta : Double) : Result[] {
    // Encode cost Hamiltonian
    using (qs = Qubit[Length(weights)]) {
        // Initialize in superposition
        ApplyToEach(H, qs);
        // Cost unitary
        for (i in 0..Length(weights)-1) {
            Rz(2.0 * gamma * weights[i], qs[i]);
        }
        // Mixer unitary
        ApplyToEach(Rx(2.0 * beta, _), qs);
        // Measure
        return MultiM(qubits => MResetZ(qubits));
    }
}
```

When submitted via `az quantum job submit`, Azure automatically routes the job to a **logical‑qubit pool**, applies surface‑code cycles, and returns the measurement array in under **2 seconds** for a 12‑qubit instance—far faster than the 30‑second latency of a classical Monte‑Carlo simulation.
